# Podium v0.1 — Test Plan and Setup Design

**Date:** 2026-04-18
**Status:** Approved design, ready for implementation plan

## Purpose

Validate the core promise of the Podium boilerplate: **anyone cloning this repo can, in a few short steps, set up any of the four boilerplate agents — and easily modify them.**

The repo currently ships four role overlays (`agent-architect`, `assistant`, `tutor`, `creator`) on a shared runtime stub. The engine prints a config summary and exits — no conversation loop, no live setup flow. This spec defines the test plan and minimal implementation that gets us from that state to "clone → `/setup` → talk to your agent" for every role.

## v0.1 Ideal End State

From a clean clone, a new user can:
1. Launch Claude Code and run `/setup`.
2. Pick one of four roles (flavors).
3. Receive a working agent that responds in character to a message.
4. Re-run `/verify` any time to confirm health.
5. Modify the agent three ways — swap role, edit identity, add a skill — and see each change take effect.

Every assumption between the current state and this ideal is covered by a failing (red) test that turns green as we implement.

## Central Metaphor: NanoClaw-Shaped Flavors

Podium is a set of flavors on top of a NanoClaw-inspired harness. We mirror NanoClaw's setup pattern — AI-native `/setup` skill, structured stdout status blocks, Claude Code as the runtime for heavy interaction. Roles are the contracts: each role materially changes identity, available skills, and domain expertise, but the harness is shared.

## Architecture

```
tests/
├── conftest.py                 # shared fixtures: tmp_repo, role param,
│                               # parse_status_block, live-mode gate
├── contracts/
│   └── role_contract.py        # ONE contract parameterized across four roles
├── l1_boot/                    # fast, mocked, always run
├── l2_setup/                   # shell + stdout blocks, no live LLM
└── l3_behavior/                # Claude Code subprocess, gated @live

setup/                          # NEW — NanoClaw-style setup module
├── cli.py                      # python -m setup [--step install|verify]
└── steps/
    ├── install.py              # deps + role + provider + active-role.yaml
    └── verify.py               # L1 boot check, emits status block

setup.sh                        # NEW — bash bootstrap (python ver, pip, log)

.claude/skills/
├── setup/SKILL.md              # NEW — /setup AI-native entry
└── verify/SKILL.md             # NEW — /verify health check (standalone)

runtime/engine.py               # MODIFIED
  - adds --message <text> CLI arg
  - assembles role context (identity + style + skill list)
  - shells to `claude -p <msg> --append-system-prompt <ctx>`
  - keeps an LLMClient protocol so LiteLLM is a drop-in later

runtime/providers.yaml          # MODIFIED — bump retired model string

agent/memory/active-role.yaml   # NEW — written by install step
```

**Key architectural decisions:**

- **v0.1 message loop shells to `claude -p`**, not LiteLLM. Claude Code is the harness. LiteLLM is deferred to when non-Claude providers matter.
- **v0.1 requires Claude Code installed and authenticated.** This mirrors NanoClaw's model. The API-key-only path arrives with LiteLLM in v0.2.
- **`LLMClient` protocol** in the engine ensures the LiteLLM swap is a class addition, not a rewrite.
- **`/setup` and `/verify` are separate Claude Code skills**. `/setup` calls `/verify` at the end (async, non-blocking), but `/verify` is standalone for re-running health checks on demand.

## The RoleContract

One parameterized test module, four roles, identical assertions. If any role fails, the output identifies which flavor broke.

```python
ROLES = ["agent-architect", "assistant", "tutor", "creator"]

@pytest.fixture(params=ROLES)
def role(request):
    return request.param
```

### Nine assertions every role must satisfy

Three per modify-surface dimension (setup, boot, behavior). Each maps to an assumption between current and ideal state.

| # | Layer | Assertion | Assumption tested | Status today |
|---|---|---|---|---|
| 1 | L1 | Role has `identity/constitution.md` and `identity/style.yaml` | Role directory complete | GREEN |
| 2 | L1 | `discover_skills(role)` returns ≥5 base skills with valid frontmatter | Role has usable skill set | GREEN |
| 3 | L1 | `resolve_active_role()` returns this role when `PODIUM_ROLE=<role>` | Role resolution works | GREEN |
| 4 | L2 | `python -m setup` exits 0 end-to-end for this role | Setup flow runs cleanly | **RED** |
| 5 | L2 | `INSTALL` status block shows chosen role; `VERIFY` shows boot success | Status reporting is structured | **RED** |
| 6 | L2 | After setup, `agent/memory/active-role.yaml` contains this role | Setup persists choice | **RED** |
| 7 | L3 | `engine.py --message "who are you?"` response contains role keyword¹ | Role identity colors output | **RED** |
| 8 | L3 | Identity edit: flip one line in `style.yaml`, response differs from baseline² | Identity is mutable and observable | **RED** |
| 9 | L3 | Skill add: drop `skills/base/ping/SKILL.md`, `--message "ping"` response contains "PING" | Skills are discoverable and usable | **RED** |

¹ `ROLE_KEYWORDS = {architect: "design", assistant: "schedule", tutor: "learn", creator: "create"}`. Case-insensitive substring match.
² `difflib.SequenceMatcher(a=baseline, b=edited).ratio() < 0.95`.

### Coverage math

9 assertions × 4 roles = **36 parameterized cases**. Roughly 12 pass today (L1), 24 fail (L2 + L3) — that is our red bar.

### NOT in the contract (deliberately deferred)

- Extension pack activation (v0.2)
- Scheduler firing (v0.2)
- Onboarding runner reading `questions.yaml` (v0.2)
- Multi-turn conversation / session memory (v0.2)
- LiteLLM provider swap (v0.2 — `LLMClient` protocol preserves the seam)

## Setup Flow

One user-facing command. Two internal steps (for tests). One async probe handled by `/verify`.

**Student experience:**
```
git clone <repo> && cd podium
claude
/setup
```

**Under the hood:**

| Step | Runtime | Responsibility | Status block |
|---|---|---|---|
| `install` | sync, <10s | Python deps, `claude` CLI check, role selection, writes `active-role.yaml` | `=== PODIUM SETUP: INSTALL ===` |
| `verify` | sync, <1s | L1 boot check for active role | `=== PODIUM SETUP: VERIFY ===` |
| probe (via `/verify` skill) | async, ~5s | Live behavior probe via Claude Code subprocess | `[probe]` log lines |

**Sample status blocks:**

```
=== PODIUM SETUP: INSTALL ===
PYTHON_VERSION: 3.11.4
DEPS_OK: true
CLAUDE_CLI: /usr/local/bin/claude
CLAUDE_AUTH_OK: true
ROLE: tutor
ACTIVE_ROLE_WRITTEN: true
STATUS: success
=== END ===

=== PODIUM SETUP: VERIFY ===
ROLE: tutor
SKILLS_CORE: 5
SKILLS_BASE: 8
BOOT_STATUS: success
STATUS: success
=== END ===
```

**Parallel behavior probe** — `/setup` launches `/verify --probe` via `Bash(run_in_background=true)` after install+verify complete. Student gets control back immediately. If probe fails, only `/verify` is re-run — each step is idempotent and independently retryable.

**Five assumptions the setup flow tests** (down from eight in earlier drafts):

1. Python 3.10+ and PyYAML work (`install`)
2. User can pick one of the four roles (`install`)
3. `claude` CLI is installed and authenticated (`install`)
4. Engine boots for the chosen role (`verify`)
5. Role responds in character to a live message (probe)

## Test Layers in Detail

### L1: Boot tests (`tests/l1_boot/`)

Mocked, fast, always run. ~20 test cases.

- YAML loaders exercised with synthetic fixture directories in tempdir.
- `resolve_active_role()` priority: env > `active-role.yaml` > default. Parameterized across roles.
- Every role has constitution, style.yaml, ≥5 base skills, all SKILL.md files have valid frontmatter.
- `providers.yaml` model string is not on a hardcoded retired-model blocklist.

### L2: Setup tests (`tests/l2_setup/`)

Subprocess-based, no network, no LLM calls. ~13 cases. `claude` CLI presence/auth checks are mocked via fixture that shims `PATH`.

- Each step runs in an isolated `tmp_repo` fixture (repo copy).
- Stdout parsed by `parse_status_block(name, stdout)`.
- `install` with each valid role → exits 0, writes `active-role.yaml`, `STATUS: success`.
- `install` with invalid role → exits non-zero, `STATUS: invalid_role`.
- `install` with `claude` CLI missing from PATH → exits non-zero, `STATUS: claude_cli_missing`.
- `install` with `claude` present but unauthenticated → exits non-zero, `STATUS: claude_auth_failed`.
- `install` idempotency: second run reports `ALREADY_CONFIGURED: true`, exits 0.
- `verify` on synthetic broken role → exits non-zero, `BOOT_STATUS: failed`.

### L3: Behavior tests (`tests/l3_behavior/`)

Claude Code subprocess, gated by `@pytest.mark.live`. ~12 cases.

- Skip if `claude` not on PATH OR `PODIUM_LIVE=0`.
- Default CI: skipped. Default local: runs if `claude` is present.
- Harness: `run_role_message(role, message) -> RoleResponse(text, latency_ms, role_keyword_matched)`.
  - Assembles role context into a system prompt string.
  - Shells `claude -p <msg> --append-system-prompt <ctx> --output-format json`.
  - Parses JSON, returns structured result.
- Three tests per role:
  - Role keyword appears in response to "who are you?".
  - Identity edit changes response materially.
  - Skill add is discovered and used.

### Shared harness (`tests/conftest.py`)

- `tmp_repo` fixture — repo copy in tempdir, Path return value.
- `parse_status_block(name, stdout) -> dict` — the single piece we port from NanoClaw's pattern directly.
- `role` fixture — parameterized across the four role names.
- `live_mode` marker + collection hook skipping L3 when `claude` unavailable.

### Test count

~20 L1 + ~13 L2 + ~12 L3 = **~45 total**, of which ~33 are red today.

## Red → Green Roadmap

Implementation order. Each step turns a specific slice of red to green and is small enough for one focused session.

1. Write L1 tests first. Harness established; most pass immediately.
2. Write the RoleContract scaffolding with L2/L3 as `xfail`.
3. Bump `providers.yaml` model string.
4. Build `setup/cli.py` `install` step.
5. Build `setup/cli.py` `verify` step.
6. Add `engine.py --message` arg; assemble context and shell to `claude -p`.
7. Write L3 harness + `@live` marker.
8. Write `.claude/skills/setup/SKILL.md` and `.claude/skills/verify/SKILL.md`.
9. Update `README.md` and `CLAUDE.md` for new three-line quick-start.

Items 1–2 ship the harness with no runtime changes. Items 3–5 make setup real. Items 6–7 make conversation real. Items 8–9 make it discoverable.

## Out of Scope / Deferred to v0.2

- Scheduler firing cron jobs
- Extension pack activation (54 extension skills dead until v0.2)
- Onboarding runner reading `questions.yaml` and writing `preferences.yaml`
- Multi-turn conversation / session memory
- LiteLLM provider swap (seam preserved via `LLMClient` protocol)
- Telegram / webhook channels
- Container isolation (NanoClaw's security model)
- `role.yaml` schema design

Any of these remaining broken or absent in v0.1 is expected.

## Success Criteria

v0.1 is complete when:

1. All ~45 tests pass when `claude` CLI is installed, authenticated, and on PATH.
2. `claude` → `/setup` → `/verify` works end-to-end for all four roles on a clean clone.
3. The three modification paths (swap role, edit identity, add skill) each produce a visible behavior change.
4. README quick-start is three lines and accurate.
5. The `LLMClient` protocol is in place so LiteLLM integration in v0.2 is an additive change.
